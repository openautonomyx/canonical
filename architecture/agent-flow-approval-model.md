# Agent-Built Flow Approval and Publishing Model

## Core Rule

Agents can build flows, but flows are published only after approval.

```text
Agent drafts.
Human reviews.
Policy validates.
Tests pass.
Approver approves.
Flow is published.
Audit records everything.
```

## Operating Model

```text
Intent
→ Agent drafts flow
→ Business logic mapped
→ Tools/connectors selected
→ Data contracts defined
→ Policy gates attached
→ Test cases generated
→ Simulation run
→ Review requested
→ Human / owner approval
→ Security / compliance approval if required
→ Publish as Flow Pack
→ Activate in workspace
→ Monitor running flows
→ Optimize
→ Audit
```

## Agent Boundary

The agent may:

```text
create flow drafts
suggest business logic
map lifecycle steps
select tool connectors
generate test cases
generate documentation
prepare approval request
prepare audit evidence
```

The agent may not:

```text
publish without approval
activate high-risk flows without approval
bypass policy
self-approve
remove audit evidence
grant itself more access
connect unauthorized tools
use data outside the employee/workspace boundary
```

## Approval Gates

```text
Flow Owner Approval
  confirms business logic and outcome

Data Owner Approval
  confirms data use, lineage, retention, and sensitivity

Security Approval
  confirms tool scope, secrets, identity, access, and risk

Compliance Approval
  confirms mandates, evidence, and retention

Workspace Owner Approval
  confirms activation in workspace
```

## Publishing Rule

```text
No flow is published unless:
  owner exists
  business logic is defined
  expected outcome is defined
  input and output contracts exist
  tool scopes are declared
  policy gates are attached
  tests/simulation pass
  approval is recorded
  audit event is written
```

## Flow Pack Object

```yaml
flow_pack:
  id: employee_onboarding
  name: Employee Onboarding Flow
  lifecycle: employee
  status: review_requested
  created_by: twin_agent_emp_123
  owner: hr_manager

  business_logic:
    objective: employee_ready_to_work
    rules:
      - manager_required
      - identity_required
      - device_required
      - access_approval_required
      - audit_required

  connectors:
    - hrms
    - sso_scim
    - glpi
    - saas_management
    - nextcloud
    - mattermost

  policy_gates:
    - access_policy_check
    - separation_of_duties
    - data_classification_check
    - approval_required

  tests:
    simulation_required: true
    test_cases:
      - normal_joiner
      - contractor_joiner
      - privileged_role_joiner

  approvals:
    required:
      - flow_owner
      - security_admin
      - data_owner
    status: pending

  audit:
    required: true
    retention: default_tenant_policy
```

## Final Law

```text
Agent builds the flow.
Business owner validates the logic.
Policy validates the boundary.
Security validates the tools.
Data owner validates the data.
Approver publishes the Flow Pack.
Audit proves the approval path.
```

## Product Statement

Canonical Work OS lets Employee Twin Agents and Fabric Agents build lifecycle flows, but every flow becomes usable only after governed review, simulation, approval, publishing, and audit.
